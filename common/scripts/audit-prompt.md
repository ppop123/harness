# 代码审计 Prompt

> **使用方式**：对 Claude 说"做一次代码审计"，Claude 会按照本文件的步骤执行。
> 建议频率：每周一次。耗时：5-10分钟。

---

## 审计指令（复制给 Claude 执行）

```
请按照以下步骤对这个项目做一次代码审计，检查与 docs/golden-principles.md 的偏差。

## 步骤 1：读取规范
先读 docs/golden-principles.md，了解本项目的黄金原则。

## 步骤 2：扫描违规

检查以下模式，并列出每个发现：

### G1 违规（类型安全）
- TypeScript: 搜索 `: any`、`as any`、`@ts-ignore`
- Python: 搜索没有类型注解的函数定义

### G2 违规（边界验证）
- 搜索 API route 中直接使用 req.body/req.json() 而没有验证的地方
- Python: 搜索直接使用 dict 参数而没有 Pydantic 模型的 route

### G3 违规（重复代码）
- 搜索看起来相似的函数（名称相近、逻辑相近）
- 特别关注跨文件的重复实现

### G4 违规（错误处理）
- 搜索空的 catch 块: `catch.*\{\s*\}`
- Python: 搜索 `except: pass` 或 `except Exception: pass`

### G5 违规（硬编码配置）
- 搜索可能的 API keys、数据库连接字符串
- 搜索 localhost、127.0.0.1 硬编码的 URL

### 层级违规（architecture.md）
- 检查 UI/组件 层是否有数据库调用
- 检查 Service 层是否有 HTTP 相关代码

## 步骤 3：生成报告

输出格式：

### 代码审计报告 - [日期]

**整体评分**: [1-10分]

**发现的问题**:

| 严重程度 | 文件 | 行号 | 问题描述 | 建议修复 |
|---------|------|------|---------|---------|
| 🔴 高   | ...  | ...  | ...     | ...     |
| 🟡 中   | ...  | ...  | ...     | ...     |
| 🟢 低   | ...  | ...  | ...     | ...     |

**本周需修复（优先级高）**:
1. [...]
2. [...]

**下周可以处理**:
1. [...]

## 步骤 4：询问是否开 PR

对于每个高优先级问题，问我："要现在修复 [问题X] 吗？"
```

---

## 审计后处理

审计完成后，Claude 会：
1. 开一个或多个 refactor PR，每个 PR 只解决一类问题
2. 每个 PR 描述里说明修复了哪条黄金原则违规
3. 大部分小修复（类型注解、空 catch 块）可以直接 automerge

> 这就是 OpenAI Harness Engineering 文章中的「垃圾回收」机制。
